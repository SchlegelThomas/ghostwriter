Intent

Close the structure-browse gaps called out in Write review: every manuscript scope must be openable with descriptions; launchpad card chips must be real Draft / Reader / Canvas actions; Characters rail must support relationship link editing (not only a list); parts get first-class descriptions like chapters.

Out of scope: freehand ink polish, OpenAI provider wiring, Playwright (until founder verifies the combined Write + structure outcome).

Acceptance criteria

  • Parts store optional summary; writers can edit it from the part launchpad; navigator shows it.
  • Chapter / scene / part / character launchpad cards expose working Draft, Reader (scenes), and Canvas actions (not decorative labels).
  • Characters rail lists cast; opening a character shows notes/sheet plus add/remove knowledge links (kinds) and linked scenes.
  • Prior Write chrome (slim Draft, sketch modal, rail timeline, pinned utilities) remains coherent.
  • pnpm verify green; browser walkthrough recorded; Playwright deferred.

Tasks

  1. Domain: ManuscriptPart.summary + part.update; capability + API contract.
  2. Storage: migration 0010 on manuscript_parts.summary; Postgres read/write parity.
  3. Navigator + launchpad: surface part summaries; description editors on structure launchpads.
  4. Launchpad entry actions: Draft / Reader / Canvas press handlers with stopPropagation.
  5. Characters browse: relationship graph panel using storyKnowledge.setKnowledgeLink + sheet/notes.
  6. Docs: PRODUCT, API, ARCHITECTURE note; record log + WHERE-I-LEFT-OFF.
  7. Tests via routine-tests; pnpm verify; founder walkthrough.

Tests and verification

  • Core: part.update summary set/clear; knowledge link add/remove unchanged.
  • Storage: migrate-from-empty + part summary round-trip.
  • UI: launchpad action helpers / structure projections.
  • Browser: open part → edit description; character → link/unlink; scene card → Draft/Reader/Canvas.

Update PRODUCT (structure browse + Characters), API (part.update, sketch detail already landed). No new ADR unless relationship editing invents a second store — reuse existing story-knowledge link commands. Living design: Mockups 3.0/4.0 structure + cast language; do not regenerate mockup files.
