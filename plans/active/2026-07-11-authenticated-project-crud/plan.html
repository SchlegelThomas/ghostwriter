Ghostwriter major delivery epic

Authenticated writing workspace, revisions, and Story Canvas

Status: tree + DAG UX accepted — autonomous feature implementation active
Branch: feat/authenticated-project-crud
Owner: founder + agent

Intent

Replace the public Bellwether fixture with Ghostwriter's first real writer-owned loop: sign in with Google, create a durable account and profile, see only projects the account may access, create a project with its first book, and manage the currently implemented project hierarchy and story-knowledge records through the responsive web client, core use cases, Hono service, and Lakebase persistence.

This is one major epic and one branch, delivered as small vertical checkpoints. “CRUD” means explicit writer-domain commands with validation, authorization, conflict handling, and safe removal semantics—not arbitrary table endpoints, JSON Patch, hidden cascades, or transport logic that bypasses packages/core.

Founder-directed expansion (2026-07-12): checkpoint a73d532 preserves the locally accepted identity and metadata kernel. Before opening its pull request, this same branch now also delivers the real writing loop from the living design: safe current-kernel CRUD closure, schema-versioned Tiptap scene bodies, server-acknowledged autosave, immutable checkpoints and variants, compare/restore history, minimal unacknowledged-work recovery, and Story Canvas state management over the same canonical scene and story IDs. Work proceeds in bounded vertical milestones with tests and truthful plan records after each slice.

Founder UX review (2026-07-12): the working milestone validates behavior but its form/card-heavy Project setup and scene metadata surfaces are rejected as the primary interaction model. The replacement proposal uses a persistent filesystem-like manuscript tree plus a directed relationship graph, layered chapter/scene drill navigation, Canvas workflow lenses, inspector-based edits, and per-action server-acknowledgement toasts. The interactive proposal and its delivery contract live in Story tree + relationship graph navigation.

Out of scope: passwords and additional identity providers; subscriptions; invitations and editor roles; comments, tracked suggestions, review requests, and real-time subscriptions/presence; same-scene multi-cursor editing and complete offline project replicas; AI/model calls and image generation; remote MCP canonical writes; import/export and account exit; irreversible authored-content purge; native freeform Canvas editing; and marketplace, publishing, rights, or studio systems. A minimal owner-only scene lease is in scope because ADR 0002 requires direct-edit protection.

Acceptance criteria

  • A new writer can choose Continue with Google, complete Google's real OAuth flow, return to Ghostwriter, and receive exactly one durable account/profile; repeating the flow resumes the same account without duplicate rows.
  • The browser resumes and revokes an opaque server-side session correctly, signs out cleanly, and never stores a Google token, provider secret, or durable session bearer in browser JavaScript storage.
  • OAuth callback URLs, trusted origins, CORS, cookies, CSRF/state handling, and production secrets are explicit and environment-specific. The primary web flow proxies /api through Cloudflare Pages to Fly so the browser uses first-party cookies rather than third-party cookies between pages.dev and fly.dev.
  • Authentication is required before every product and onboarding surface in this epic. No signed-out Spark, fixture workspace, or locally saved project state is exposed.
  • Signed-out requests receive 401; authenticated writers receive 404 or 403 without leaking another account's project metadata; every project query and mutation enforces project access in shared policy before storage changes.
  • An authenticated writer sees a real project library, can create a project and first book, reload the browser, and reopen the same server-acknowledged Lakebase state with no fixture fallback.
  • A project owner can rename, archive, and restore a project. Archived projects leave the normal library and remain recoverable; permanent purge is unavailable in this epic.
  • A project owner can create, rename, change status, and reorder books while the domain continues to require at least one book and refuses unsafe or ambiguous removal.
  • A project owner can create, rename, reorder, and safely remove empty manuscript parts and chapters; create, edit metadata, move, reorder, archive, and restore scenes; and preserve one canonical manuscript order.
  • A project owner can create, edit, link, unlink, archive, and restore the kernel's story-knowledge records without creating cross-project, dangling POV, or dangling scene references.
  • Named editions remain read-only until immutable scene/project revisions exist; the UI and API do not imply that edition history is implemented.
  • Every mutation accepts an expected project version (or equivalent explicit precondition), applies atomically, and returns a stable conflict response without partial effects when the client's base is stale.
  • The web client shows Saving, Saved, validation, authorization, conflict, retryable failure, empty, and archived states truthfully. Saved appears only after durable backend acknowledgement.
  • Wide and narrow web layouts support the account gate, project library, creation flow, navigation, and core editing controls with keyboard operation, visible focus, semantic labels, and no hover-only action.
  • The capability registry identifies each new core use case and its web/backend binding. MCP read parity uses real core queries where authorization is available; canonical MCP mutations remain an explicit exception until the agent-authority and remote-authorization ADRs are accepted.
  • Hermetic browser and integration tests cover account bootstrap, session gating, project creation, representative update/archive/restore flows, stale writes, sign-out, and cross-account isolation. A manual acceptance run covers the real Google provider because CI does not automate a personal Google login.
  • Forward-only Drizzle migrations reproduce identity, profile, ownership, archive, and concurrency schema from empty in PGlite and on the PR's Lakebase branch. pnpm verify, the web export, and the selected end-to-end suite pass before handoff.
  • The current-kernel UI exposes truthful scene ordering/placement, proactive archive and safe-removal refusals, confirmation for destructive-looking actions, profile conflict recovery, and representative wide/narrow browser coverage.
  • Each scene owns one schema-versioned ProseMirror JSON document with stable top-level block IDs. Core remains free of DOM, React, and Tiptap types; metadata and manuscript order remain separate canonical records.
  • Scene-body writes use a dedicated expected scene revision rather than the coarse project version. A visible owner lease guards direct editing; stale or expired writes apply nothing and open recovery/compare instead of silently overwriting prose.
  • The Draft workspace debounces prose saves, shows Saving during an in-flight write, shows Saved to project only after durable acknowledgement, and reopens the last acknowledged document unchanged.
  • A minimal browser recovery buffer preserves only unacknowledged scene text across refresh or tab failure, is clearly not offline project storage, and clears after acknowledgement or sign-out.
  • Immutable scene revisions record schema version, content hash, parent, actor, origin, reason, and timestamp. Writers can create a checkpoint and named variant, compare two revisions, and restore prior work as a new attributable revision without rewriting history.
  • Story Canvas boards, objects, regions, typed links, notes, and imagery metadata reference canonical scene/story IDs and never copy prose. Writer-created links are confirmed; deterministic ghost objects remain provisional until confirmed or dismissed.
  • Canvas-created scenes require explicit manuscript placement. The reading-order spine derives from canonical manuscript order, exposes spatial drift, and never silently reorders Draft.
  • Wide web supports Draft, Canvas, and split view with shared scene selection. Narrow web provides keyboard-accessible reading, review, capture, and ordered Canvas actions rather than a compressed freeform board.
  • Scene-body, history, lease, and Canvas capabilities are owner-authorized, use dedicated optimistic preconditions, persist through memory and Postgres contracts, and are registered with explicit MCP mutation exceptions until agent authorization exists.
  • Migration fixtures, editor schema/IME/paste/keyboard checks, large-scene and 500-object Canvas smoke tests, and wide/narrow Playwright journeys cover write → checkpoint → compare/restore → Canvas handoff before PR handoff.
  • Wide Draft/Canvas/Split use a persistent manuscript tree rather than horizontal scene chips or standalone card/form setup. Project → Book → Part → Chapter → Scene remains visible, searchable, keyboard-navigable, and authoritative for manuscript order.
  • Canvas combines the structure tree with a directed relationship graph and workflow lenses for Outline, Relationships, Continuity fixtures, Plan → Draft, and Review. Lenses change projection and emphasis rather than creating new canonical stores.
  • Chapter and scene nodes support layered drill-in/drill-out with breadcrumbs, Back/Escape, focus management, retained selection/viewport, a 400–500 ms camera transition, and an instant reduced-motion fallback.
  • Metadata edits move to inline tree rows or the selection inspector. Primary flows do not require separate “Save part/chapter/scene” forms; destructive actions keep explicit confirmation.
  • Every structure, metadata, Canvas, archive/restore, checkpoint, conflict, and recovery action receives truthful acknowledgement UI. Success toasts appear only after durable server acknowledgement and expose Undo/Restore only when safe.
  • Narrow web uses book tabs, an ordered structure/relationship view, one focused Draft scene, and decision-first review actions. It does not compress freeform spatial drag into a phone layout.
  • Book Reader is a read-only projection over canonical manuscript order and acknowledged scene documents. Wide web provides bound spreads and chapter/page navigation; narrow web provides continuous chapter landmarks. Reader takes no editing lease and creates no prose copy.
  • Reader opens from tree, Draft, Canvas, Split, or a scene lens at the same canonical scene ID and returns to the exact prior mode, scene, tree expansion, drill scope, Canvas selection, and focus target.
  • Reader exposes scene-linked story/Canvas context through an optional rail. Inline passage links remain unavailable until a stable block-pin model is separately accepted; the UI does not fake anchors by text matching.
  • Wide Split panes are resizable and preserve Draft mount, caret, and lease while both panes remain visible. Medium layout collapses the tree/inspector deliberately; narrow layout uses mode switching rather than split.
  • Storyboard-first creation, write-first Canvas placement, tree scene move/reorder, plan-to-Draft, scene-to-Reader, and Reader return all preserve one scene ID, explicit manuscript authority, version preconditions, and acknowledged state.

Risks and decisions

  • Accepted identity stack: self-host Better Auth on the existing Hono backend, use its Drizzle/Postgres adapter and Google provider, and keep Google as the only login method in this epic. Managed auth adds a new data processor/cost boundary; custom OAuth/session code adds avoidable security risk.
  • Accepted web session topology: proxy /api through the Cloudflare Pages origin to Fly. Direct credentialed cookies between pages.dev and fly.dev are third-party and fail in privacy-restrictive browsers.
  • Accepted entry boundary: require Google sign-in before entering any product or onboarding surface. A temporary pre-auth Spark is not part of this epic.
  • Accepted deletion posture: archive/restore authored records and refuse ambiguous structural deletion. Permanent project/account purge waits for export, retention, backup, and account-exit policy so this epic does not violate writer ownership.
  • Expanded scope boundary: by founder direction, the current branch now includes safe metadata CRUD closure, prose bodies, owner leases, personal revision history/recovery, and Story Canvas state management before PR. Editions publishing, comments, collaborators, real-time subscriptions, AI, import/export, and purge remain later outcomes.
  • Accepted concurrency split: keep project metadata on project.version, prose on a per-scene working version, and spatial state on a Canvas board version. Mixing debounced prose, Canvas drag, and low-frequency structure edits behind one project counter would create false conflicts and oversized aggregate writes.
  • Required persistence refactor: the current Postgres command adapter deletes and rebuilds child rows. Prose/history foreign keys make that unsafe, so metadata persistence must become surgical before revisions attach to scenes.
  • Accepted revision model (ADR 0006): store canonical versioned ProseMirror JSON with stable top-level block IDs, mutable acknowledged working state plus append-only content-addressed revisions, single-parent checkpoints in this epic, restore as a new revision, block-aware comparison, and no keystroke event sourcing or auto-merge.
  • Accepted recovery model (ADR 0006): use a bounded origin-private IndexedDB buffer only for unacknowledged prose, explicitly disclose its local-at-rest limits, expire it, and clear it after acknowledgement/sign-out. It is not a second canonical store or offline mode.
  • Accepted Canvas model (ADR 0007): one project board stores versioned object placement and typed links referencing canonical IDs; manuscript order alone drives the spine, and an accessible ordered projection exposes every canonical action without drag.
  • Transport boundary: owner-only HTTP leases and optimistic revision checks are in scope. Live subscriptions, presence, invitations, and reconnect transport remain open; product copy must not imply they ship in this epic.
  • MCP: exposing direct canonical author CRUD to an unauthenticated local fixture server would violate the accepted agent authority direction. Read bindings may advance; mutation parity is documented as a temporary security exception until scoped grants and remote/local authorization are accepted.
  • OAuth environments: Google requires exact callback URLs and does not support wildcard branch previews. Local, one stable acceptance environment, and production need deliberate registration; required CI uses a deterministic test identity boundary.
  • Accepted credential-rotation deferral: an existing backend service credential was entered directly in local shell history. Its value was not copied. The founder chose to rotate it later; this accepted residual risk is recorded rather than silently treated as resolved. All new auth secrets use stdin/secret stores.
  • Epic size: the founder explicitly prefers one large pre-PR branch and iterative autonomous delivery. Preserve reviewability through milestone commits, additive migrations, bounded modules, immediate records, and green verification at every checkpoint. Stop only for a genuine product, architecture, security, cost, data-loss, visual-review, push/PR, or deployment gate.
  • UX direction under review: tree structure remains canonical, DAG relationships remain interpretive, drill depth remains view state, and toasts remain acknowledgement UI. Persisted metadata undo or canonical drill scope would require a later architecture decision; this proposal does neither.
  • Accepted UX direction: on 2026-07-12 the founder accepted the persistent tree + DAG, layered drill, workflow-lens, narrow, and toast proposal and authorized implementation.
  • Reader boundary: v1 Reader uses acknowledged working heads and scene-level linked context. Stable inline passage pins and edition/project snapshot pinning require their own accepted model and must not be simulated with text matching.
