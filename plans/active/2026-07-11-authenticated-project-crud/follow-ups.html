Candidate B · after authenticated CRUD

Scene editor and revision history

Suggested branch: feat/scene-editor-revision-history

Intent

Turn scene metadata into a real writing surface: persist schema-versioned ProseMirror JSON with stable block IDs, edit it through Tiptap, acknowledge durable writes, and preserve immutable checkpoints and named variants so authors can write, compare, restore, and recover without fear.

Required outcomes

  • One canonical ProseMirror document per scene, validated and migrated through platform-neutral editor codecs.
  • Tiptap on web with a narrow Expo DOM-component contract reserved for native; no DOM or Tiptap type enters core.
  • Debounced working-state writes guarded by a visible scene lease and expected scene revision; Saved only follows server acknowledgement.
  • Minimal encrypted browser recovery for unacknowledged text, cleared after acknowledgement and never presented as offline project access.
  • Immutable checkpoints at meaningful boundaries, content hashes, actor attribution, restore as a new revision, named variants, compare, and a revision-aware undo path.
  • Scene tree create/move/archive behavior from the active epic remains canonical; editor bodies never create a second manuscript order.
  • UI and MCP use the same read/revision/proposal core capabilities. Direct external-agent canonical writes remain prohibited.

Decisions and documentation

Create the queued editor/revision ADR covering ProseMirror schema, stable block IDs, working state, checkpoint graph, content hashing, lease/revision semantics, retention, and recovery expiry. Reconcile roadmap candidates 1.3 and 1.4 into one plan only if the plan can still deliver and verify a coherent vertical outcome.

Verification gate

  • Create, write, close, reopen, reconnect, conflict, compare, restore, and undo scenes without losing an acknowledged edit.
  • Schema round trips, migration fixtures, IME/paste/selection/keyboard behavior, large-scene performance, and crash recovery pass.
  • A stale write changes nothing and opens comparison/recovery; all checkpoints are attributable and content-addressed.

Explicit non-goals

Same-scene multi-cursor editing, full offline replication, comments/suggestions, AI prose generation, import/export, and Story Canvas freeform editing.

Candidate C · after authenticated CRUD and editor foundations

Story Canvas foundation

Suggested branch: feat/story-canvas-foundation

Intent

Implement the living Story Canvas 2.0 direction as a spatial view over the same authenticated project, scenes, and story records: writers can begin from a board or from prose, arrange ideas visually, and move between Canvas and Draft without duplicated truth or silent model authority.

Required outcomes

  • Canvas objects and typed links reference canonical scene/story IDs; placement, regions, notes, and imagery metadata do not copy scene bodies.
  • Write-first and storyboard-first journeys converge on the same scene records. Creating a scene from a Canvas region states its manuscript placement before applying it.
  • The reading-order spine derives from canonical manuscript order, shows drift, and never silently reorders Draft.
  • Writer-created pins/links are solid; inferred cards, links, beats, and imagery are dashed/provisional until reviewed. No AI implementation is required for deterministic fixture proposals.
  • Canvas, Draft, and split view preserve selection and link context across wide web; narrow web is read/review/capture focused rather than freeform spatial editing.
  • Every Canvas mutation is authorized, expected-version checked, server-acknowledged, undoable through accepted history primitives, and represented in the capability registry.

Decisions and documentation

Extend or supersede the domain-boundary ADR only if Canvas ownership changes. Specify the spatial data model, viewport/placement persistence, typed link lifecycle, attachment/image boundary, large-board indexing, accessibility alternative, and scene-handoff transaction before implementation. The living design remains the visual and interaction source and must not be regenerated by the delivery plan.

Verification gate

  • Canvas and Draft always resolve the same scene IDs across create, rename, move, archive, restore, split view, and reload.
  • Empty, seeded, populated, region-selected, provisional, stale-link, spine-drift, read-only recovery, and 500+ object states are usable and measured.
  • Keyboard and screen-reader users can navigate an equivalent ordered representation and complete every canonical action without spatial drag gestures.

Explicit non-goals

Image generation, automated manuscript derivation, real model calls, mobile freeform Canvas editing, marketplace assets, and simultaneous same-object co-editing.
