Founder review · UX proposal

Story tree + relationship graph navigation

Status: accepted 2026-07-12 — implementation active
Branch: feat/authenticated-project-crud
Parent plan: Authenticated writing workspace, revisions, and Story Canvas

This plan supplement responds to the 2026-07-12 milestone review. It does not open a second branch or replace the active epic. The interactive visual proposal is appended to Ghostwriter Mockups 2.0.

Intent

Replace the form-and-card “Project setup” experience with a writer-facing structure navigator that combines a filesystem-like manuscript tree with a relationship DAG. The tree remains the canonical hierarchy and reading order; the graph reveals meaning, threads, dependencies, continuity, and provisional ideas without becoming another source of manuscript order.

Make Canvas a workflow surface rather than an object-management screen. Writers should be able to shape an outline, inspect relationships, move from plan to prose, review continuity, and restore prior Canvas states through one layered environment. Clicking a chapter or scene dives into that scope with an animated camera transition and an explicit way back up.

Every acknowledged action receives clear, contextual save confirmation. Persistent status stays quiet; per-action toasts explain what saved and expose Undo/Restore where the domain supports it.

Acceptance criteria

  • Wide Draft, Canvas, and Split modes share a persistent Project rail, manuscript tree, center work surface, and selection inspector. The current horizontal scene chips and primary card/form setup flow are absent.
  • The manuscript tree exposes Project → Book → Part → Chapter → Scene plus explicit Unassigned scenes. Expand/collapse, inline rename, add, reorder, move, archive, and restore actions have pointer and keyboard paths.
  • The tree is authoritative for manuscript order. Relationship edges, Canvas coordinates, regions, and story-order hints never reorder Draft without a separate explicit confirmation.
  • Canvas offers workflow lenses for Outline, Relationships, Continuity, Plan → Draft, and Review. Each lens has a clear entry, primary action, empty state, and return to neutral Canvas.
  • Clicking or pressing Enter on a chapter region/node animates a 400–500 ms zoom into a chapter lens. Breadcrumbs, Back, Escape, and reduced-motion behavior all return to a parent scope without losing selection or viewport.
  • At chapter depth, sibling context dims but remains legible; scenes, inbound/outbound links, provisional findings, and chapter-level actions become primary. Scene depth can open a Draft preview or Split view using the same canonical scene ID.
  • Persistent topbar status distinguishes Saving, Saved to project, retryable failure, conflict, and recovery. Action toasts appear only after durable acknowledgement and identify the action that completed.
  • Structure and Canvas acknowledgement toasts expose Undo where an inverse or Canvas snapshot exists. Archive acknowledgement exposes Restore. Draft saves avoid toast spam and rely on quiet status plus one post-idle acknowledgement.
  • Conflict and recovery toasts remain until resolved, do not steal editor focus, and expose explicit Reload/Use server/Keep local/Recover/Discard actions as appropriate.
  • Narrow web uses book tabs, an ordered structure/DAG view, one focused Draft scene, and decision-first review actions. It does not claim freeform spatial drag.
  • Every spatial workflow has an ordered keyboard/screen-reader equivalent; authority, stale state, relationship direction, and hierarchy depth are expressed in text rather than color or line style alone.
  • The proposal preserves ADRs 0002, 0003, 0006, and 0007: auth-first server authority, one canonical scene model, revision-safe prose, dedicated Canvas versioning, and explicit manuscript-order authority.

Interaction model

Tree = structure

  • Single click: select a row and populate the inspector.
  • Chevron / ArrowRight: expand a folder. ArrowLeft: collapse or move to its parent.
  • Enter: open a scene in Draft or dive into a chapter lens from Canvas.
  • F2 / inline title: rename and commit on Enter or blur.
  • Drag or Alt+Arrow: reorder within a parent. Move-to commands use a searchable destination menu.
  • Context menu: add child, duplicate metadata, move, archive/restore, or inspect history. Destructive actions keep the existing confirmation contract.

Graph = relationships

  • Scene, story-knowledge, note, image-reference, and region nodes always reference canonical IDs or Canvas-owned records.
  • Directed link arrows expose inbound/outbound meaning and link kind. Confirmed and provisional links retain explicit labels.
  • Workflow lenses filter/emphasize the same board rather than navigating to separate data stores.
  • The reading-order spine stays visible in Outline and Plan → Draft, with drift explained as a review state rather than an automatic reorder.

Drill stack

  1. Project lens: books, parts/chapters, cross-book threads, and global story knowledge.
  2. Chapter lens: chapter scenes, local relationships, setup/payoff flow, continuity findings, and “Open in Draft.”
  3. Scene lens: selected scene, linked knowledge/evidence, brief, history, and Draft preview.

The drill stack is UI state, not a new route or persistence authority. Canvas viewport preference records the last scope/viewport only after the writer settles on it.

Canvas workflow contracts

  • Outline · Shape the book: add or move structure from the tree, storyboard a scene with explicit placement, inspect spine drift, and choose whether to apply a proposed order through a separate manuscript command.
  • Relationships · See threads: select a relationship/thread, highlight its directed chain, inspect inbound/outbound links, and confirm/dismiss provisional edges.
  • Continuity · Check the world: scope deterministic fixture findings to the current chapter, pin evidence to scenes, and send unresolved items to Review. Real model calls remain out of scope.
  • Plan → Draft · Write this beat: select a planned scene, edit its brief/POV/status in the inspector, open Split, and carry the same selection into Tiptap.
  • Review · Decide what changed: browse Canvas snapshots, compare authority/stale states, Undo the latest command, or restore a prior snapshot without changing Draft prose.

Save acknowledgement contract

  • Quiet persistent status: Saving… → Saved to project; Canvas and Draft surface their own domain state while an aggregate “All changes saved” appears only when all domains are idle.
  • Metadata toast: “Chapter renamed” / “Scene moved to Game Day” with Undo when an inverse command is safe.
  • Canvas toast: “Relationship linked” / “Chapter lens updated” with Undo using the Canvas snapshot history.
  • Archive toast: “Scene archived” with Restore after the confirmed request succeeds.
  • Draft acknowledgement: show the first “Draft saved” after an idle writing burst, then suppress repeated toasts for 30 seconds; History remains the undo model.
  • Sticky problem toast: conflict, retryable save failure, expired lease, or recoverable draft remains visible until the writer chooses an explicit action.

Toasts use role="status"/aria-live="polite"; conflicts use role="alert". They never steal focus from Tiptap or a tree rename.

Tests and verification

  • Tree: expand/collapse, inline rename, add/move/reorder, archived filters, destination refusal, one canonical scene order, and keyboard parity.
  • Drill: project → chapter → scene → parent, viewport preservation, Escape, breadcrumb focus, reduced motion, and stale selected-node recovery.
  • Workflows: each lens changes only projection/filter state unless an explicit canonical command is acknowledged.
  • Toasts: no pre-ack success, action copy, stacking limit, timeout/hover pause, Undo/Restore, conflict persistence, screen-reader announcements, and no editor focus theft.
  • Responsive: four-region wide shell, collapsible medium navigator, ordered narrow view, no horizontal overflow, and no narrow freeform-drag claim.
  • Regression: Draft/history/recovery, Canvas snapshots, atomic storyboard handoff, owner authorization, and all current repository/API suites remain green.

Documentation and ADR impact

This is a UX binding change over accepted domain and persistence boundaries. No new ADR is required unless implementation adds persisted metadata undo or makes drill scope canonical. Update the active plan, record log, living design, product experience notes, and browser acceptance. Do not rewrite ADRs 0003/0006/0007.

The founder explicitly authorized adding this direction to the living design and accepted it on 2026-07-12. Existing founder-authored screens remain unchanged; the additive section now governs implementation alongside ADRs 0002/0003/0006/0007.

Risks and decisions

  • Accepted constraint: the filesystem metaphor must not expose raw files or create a second local canonical store.
  • Accepted constraint: DAG position/order is interpretive; manuscript order changes only through explicit structure commands.
  • Open review: whether single-click or double-click is the primary chapter dive gesture. Keyboard Enter is always supported.
  • Open review: whether metadata fields commit on blur immediately or use a short dirty-state Apply affordance for multi-field scene edits.
  • Open review: how much relationship detail appears on the spatial board versus the inspector to avoid visual overload.
  • Scope: deterministic continuity/derive fixtures may illustrate workflows; AI inference, collaboration presence, and timeline implementation remain later.
