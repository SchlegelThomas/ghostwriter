Architecture direction

Technical blueprint

A web-first collaborative story platform built around multi-book projects, scene-sized documents, a prose-aware version graph, shared Story Canvas objects, and one accountable proposal path for humans and agents.

Status: server-authoritative topology accepted in ADR 0002, Lakebase/Hono backend implemented in ADR 0004, Google accounts/project access implemented in ADR 0005, and owner-only Draft/history/recovery plus Canvas foundations implemented under ADRs 0006/0007; real-time/editorial transport remains open
Committed constraints preserved: Tiptap, platform-neutral core, writer ownership, and MCP parity

Architecture stance

Build one trustworthy content engine beneath a live web workspace: rich local interaction, shared project state, immutable history, prose-aware variants, and deterministic authority over every human or agent change.

  • Modular monolith: packages provide clear boundaries inside one TypeScript codebase; there are no microservices, plugin runtime, accounts, or marketplace abstractions in the writing kernel.
  • Ports and adapters: core use cases depend on storage, clock, ID, model, export, and policy interfaces. Platform packages select concrete adapters.
  • Shared capabilities: UI and MCP call the same core commands and queries. A transport never reaches around core into storage.
  • Deterministic authority: models may analyze and propose. Only validated core transitions can create canonical revisions.
  • Web-first, not web-fragile: the browser is primary; server-acknowledged edits are durable, unacknowledged input receives minimal crash/reconnect recovery, and complete export protects account exit.
  • Server authority is explicit: the shared service owns canonical project state; browser storage is limited to recovery for unacknowledged work and is not a complete offline replica.
  • Editorial real-time, not same-scene co-editing: project updates, presence, comments, tracked suggestions, variants, and review are live, while one collaborator directly edits a scene body at a time.
  • Evolution by evidence: lexical retrieval precedes embeddings; scene leases and revision checks precede CRDT editing; full offline and multiple cursors require later proof and an ADR.

Dependency rules

  • core imports no React, DOM, Electron, Node, Tiptap, provider, database, or MCP SDK types.
  • editor, storage, and ai may depend on core contracts; core never depends on them.
  • ui depends on core-facing view models and editor interfaces; platform setup remains in applications.
  • Platform adapters use explicit subpath exports such as @ghostwriter/storage/web and /node so Metro never follows native Node dependencies.
  • Electron renderer has no Node access. Files, SQLite, keychain, and export run through a narrow validated IPC API with context isolation and sandboxing.

Recommended composition: a plain-function factory such as createGhostwriterServices(dependencies) exposes typed use cases. It is easy to test with memory repositories and avoids a framework-specific dependency-injection container.

Editor, documents, and revisions

Canonical scene document

  • Store versioned ProseMirror JSON, not HTML, as the canonical scene body.
  • Assign stable IDs to addressable top-level blocks so evidence and durable proposals are not based only on fragile character offsets.
  • Keep the editor schema and migrations in packages/editor; expose DOM-free validation and codec entry points separately from React components.
  • On native, keep transaction-heavy Tiptap state inside the DOM component and bridge coarse commands, selection events, and debounced checkpoints—not every ProseMirror transaction.

Working state and history

Live working state→ Immutable checkpoint→ Variant / review branch→ Compare and combine→ Named edition

Create immutable revisions at meaningful boundaries: manual checkpoint, sustained idle, review request, import, restore, migration, before and after proposal application, before an AI run, variant creation/merge, and edition/export. Deduplicate identical checkpoints by content hash. Do not event-source every keystroke.

Model history as a content-addressed revision graph: a normal revision has one parent; an explicit combine may have multiple parents and a recorded resolution. Human-readable variant names point to graph heads. Every revision records schema version, content hash, actor, origin, timestamp, reason, and related review/proposal/run/import. The rich UI says “variant,” “compare,” “combine,” and “restore,” not branch, commit, rebase, or merge conflict.

Live collaboration state and durable content history are separate concerns. V1 uses one visible active-editor lease per scene plus expected-revision checks; comments and tracked suggestions target immutable revisions. A CRDT is deliberately deferred. Agent variants still target an immutable base and never silently rebase.

Persistence, synchronization, and portability

ADR 0002 replaces the earlier SQLite-only/local-first recommendation for v1. A responsive live web product with invited editors uses a server-authoritative shared project, explicit identity and authorization, real-time subscriptions, revision-checked writes, and minimal browser recovery.

Accepted v1 behavior

  • Canonical store: the shared service owns project heads, permissions, checkpoints, variants, comments, suggestions, reviews, and editions.
  • Save semantics: clients may render optimistically, but show Saved only after durable server acknowledgement.
  • Browser recovery: keep only unacknowledged text/commands long enough to survive refresh, tab crash, or brief disconnect; clear them safely after acknowledgement.
  • No offline promise: a disconnected user cannot open and edit the complete project in v1. Recovery mode must not be marketed as offline editing.
  • Scene coordination: one active direct editor per scene, visible presence/lease, expected revision on write, and compare/recovery on conflict.

Keep repository contracts independent of the selected backend. The shared database, identity system, real-time transport, and recovery-buffer technology are selected through a focused spike. Migrations require backups, compatibility checks, reconnect testing, and safe refusal when clients and project schemas are incompatible.

The live database is not the interchange contract. A versioned logical archive preserves project/book IDs, revision graph, Canvas placements, comments, provenance, and unknown forward-compatible fields while excluding credentials, caches, telemetry, and diagnostics. Manuscript exports remain readable projections. Web access is the primary portability experience, but platform exit cannot depend on a subscription or a running sync service.

Context engine

Project awareness should begin with deterministic, inspectable assembly—not a vector database.

  1. Resolve the workflow and writer-selected scope.
  2. Include the current selection or scene and its immutable revision.
  3. Add structural neighbors when the workflow needs them.
  4. Add explicitly linked entities, confirmed/planned facts, timeline events, threads, and voice references.
  5. Add bounded lexical search candidates from a derived index.
  6. Rank, deduplicate, trim to the workflow budget, and create a context receipt.

Each context item records artifact ID, revision, block anchor, content hash, inclusion reason, priority, token estimate, and truncation. The writer can inspect the receipt before content leaves the device.

Embedding gate: add optional local embeddings only when evaluation on representative long projects proves explicit links and lexical search miss important context. Embeddings remain derived, disposable, revision-bound, and local by default; no hosted vector store is needed for the authoring core.

AI boundary and orchestration

packages/ai adapts providers; packages/core decides allowed context, workflow policy, and whether a typed result can become a proposal.

interface ModelGateway {
  run(
    request: ModelRequest,
    signal: AbortSignal
  ): AsyncIterable<ModelEvent>;
}

The normalized contract supports streaming, validated structured output, usage, finish reason, cancellation, and typed errors. Provider SDK types never leak into core. Capability flags express provider differences instead of pretending every model supports the same behavior.

Start with bounded workflows

  • Extract candidate story facts with evidence.
  • Critique one scene against its stated intent.
  • Propose a revision to a selected passage.
  • Suggest several next-beat options.
  • Run a scoped continuity pass.
  • Create a scene or chapter summary as derived context.

Each workflow declares context categories, read-only internal tools, output schema, model capabilities, token/cost/time/step limits, cancellation behavior, and allowed proposal operations. A general autonomous planner and self-extending tool loop are deliberately out of scope.

Credential decision by surface

  • Desktop: operating-system keychain.
  • Mobile: secure platform storage.
  • Local MCP: environment or OS-protected profile.
  • Web: no durable browser secret should be described as secure. Choose session-only BYOK, provider-supported ephemeral credentials, local companion, or a hosted proxy in an ADR.

A hosted proxy changes cost, abuse, privacy, compliance, and operational boundaries. It is a product decision, not a hidden implementation convenience.

One proposal path

Capture base revisions→ Assemble context receipt→ Run bounded workflow→ Validate typed result→ Create proposal→ Review→ Atomic apply + revision

Proposal operations are domain-specific: replace a scene document, update scene metadata, add a candidate fact, create an entity, reorder scene references, or add a review finding. Avoid arbitrary JSON Patch, generic CRUD, and raw ProseMirror steps as the only durable representation.

Apply revalidates permission, schemas, proposal hash, and every base revision in one transaction. It is idempotent, creates before/after checkpoints, links provenance, and leaves the proposal reviewable. Selecting only some operations creates a new child proposal, preserving an unambiguous history.

MCP topology and parity

A shared web project changes the earlier desktop-only MCP assumption. MCP may connect either to an explicitly selected local project replica or to the authenticated shared project service, but both paths must execute the same core capabilities and project-scoped policy.

  • Local stdio remains useful for private desktop workflows and agent tools running on the writer’s machine.
  • Remote access to the shared project requires an explicit OAuth/authorization design, revocable grants, resource scoping, egress disclosure, and audit—not reuse of a browser session token.
  • Capability parity means the same core use case exists and is appropriately bound. It does not grant an external client the first-party writer’s authority.
  • MCP paths are project-scoped resource URIs; protocol inputs never expose or accept arbitrary filesystem paths.

See Agent contracts for resources, tools, permissions, provenance, and parity tests.

Collaboration evolution

Accepted v1

Shared project and editorial real-time

Globally unique IDs, author/editor roles, comments, tracked suggestions, review requests, presence, live project updates, one active direct scene editor, immutable checkpoints, minimal unacknowledged-work recovery, version compare, and complete export.

Explicitly deferred

Same-scene live co-editing

Evaluate Yjs only if real usage justifies simultaneous multi-cursor editing. Any later CRDT working state still checkpoints valid ProseMirror JSON into the immutable revision graph.

Later depth

Offline replication and larger teams

Add full offline editing, cross-device conflict support, organization roles, approval workflows, and broader CRDT coverage only when the launch topology and real usage justify them.

ADR 0002 fixes authority and collaboration depth. ADR 0004 selects the Lakebase/Hono runtime, ADR 0005 selects identity/access, ADR 0006 now implements scene prose, leases, personal history, and bounded recovery, and ADR 0007 now implements the project Canvas board, typed links, spine/drift, snapshots, and accessible ordered view. Real-time transport, collaboration roles/invitations, comments/suggestions, backup/retention policy, permanent deletion, export/account exit, and remote MCP authorization remain open.

Industry and marketplace boundary

Future publisher, studio, showrunner, rights, catalog, discovery, payment, and streaming systems belong in separate cloud bounded contexts. The private shared project service remains the authoring authority. Publishing creates an explicit, consented, revision-pinned projection or copy.

The test is separation: removing marketplace or industry services must not prevent private project writing, reading, history, or export through the core shared project service.

Privacy, diagnostics, and security posture

  • A network connection and authenticated shared service are required for normal v1 project access. Recovery copy must never imply otherwise.
  • Server access to project content is limited by documented author/editor roles, audited service operations, encryption, retention, deletion, backup, and account-exit policy.
  • Every AI call has a visible provider and context receipt; project access does not automatically grant separate model-provider egress.
  • Structured diagnostics record IDs, adapter/workflow, duration, usage, and error category—never manuscript text, raw prompts, provider output, or secrets by default.
  • Remote telemetry and crash reporting are opt-in, content-free, documented, and locally previewable before diagnostic export.
  • Imported text and model output are untrusted data. Strip active HTML, never auto-fetch embedded URLs, validate archive paths and sizes, and render safely.
  • Provider calls are mocked in required CI. Test fixtures use original synthetic fiction, not private user work or copyrighted novels.

ADR queue and decision timing

  1. Multi-book project, Story Canvas, and truth states: canonical ownership is accepted in ADR 0003; project-board spatial state, links, authority, spine behavior, and accessibility are accepted in ADR 0007.
  2. Editor document, stable block IDs, revision graph, variants, provenance, leases, and recovery: accepted in ADR 0006 before production Tiptap persistence.
  3. Server authority, minimal browser recovery, and editorial collaboration: accepted in ADR 0002; the implementation spike must select concrete services and protocols.
  4. Identity and initial project access: accepted in ADR 0005 for Google accounts, profiles, owner memberships, and first-party sessions. Editor roles, invitations, and remote MCP access remain later decisions.
  5. Agent authority and MCP mutation policy: before a real model or external client can create actionable proposals.
  6. AI provider, credentials, egress, and web boundary: before shipping provider calls on any surface.
  7. Full offline replication and same-scene co-editing: only when validated demand justifies replacing the v1 lease/revision model.
  8. Remote MCP, organizations, publication, rights, or marketplace: one or more later ADRs; none should be pre-decided inside core authoring packages.
