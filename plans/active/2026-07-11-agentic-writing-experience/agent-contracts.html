Agent integration, safety, and evaluation

Agent contracts

Assistants work in tandem with authors and editors: they receive bounded assignments, produce inspectable ideas or prose variants, and help hold entire books and series in view without becoming invisible co-owners.

Permanent authority rule

A model may read an approved context and submit noncanonical ideas, edits, or multiple variants. It never becomes the actor that applies canonical text or story facts.

Canonical change occurs only when an author—or an editor explicitly granted apply authority—reviews an exact revision-bound proposal, approves its content hash, and the deterministic core revalidates permission and preconditions before an atomic apply. The result creates a revision, provenance, audit events, and an undo path.

Author or editor assigns→ Policy scopes→ Agent reads→ Agent proposes→ Human compares and reviews→ Core applies

What “agentic” means in Ghostwriter

  • Project-aware: an agent can use permitted scenes, characters, facts, timeline, threads, voice references, and prior decisions without repeated copy/paste.
  • Goal-directed: assignments name an outcome, scope, constraints, output types, and completion condition—not just a conversational prompt.
  • Tool-using but bounded: workflows may call narrow read-only project queries under externally enforced limits.
  • Artifact-producing: agents return findings, questions, memos, candidate facts, scene cards, drafts, or typed diffs that live in the project Inbox.
  • Variant-capable: an assistant can offer several prose directions against one immutable base so authors and editors can compare or combine them without disturbing the working draft.
  • Accountable: every result shows its source scope, evidence, origin, provider, workflow version, status, and effect before acceptance.
  • Interruptible: runs expose queued, running, needs-input, ready, failed, canceled, stale, and conflicted states.

Not in v1: open-ended autonomous swarms, arbitrary shell/filesystem/network tools, background runs without visible budgets, self-granted permissions, automatic canon updates, or a general agent plugin marketplace.

Agent roles are reusable workflow lenses

Story architect

Shape and stress-test

Premise options, dramatic questions, structural views, missing turns, setup/payoff, and alternate scene paths. Produces plan artifacts, never a secret outline.

Character coach

Protect agency and change

Interviews, desire/pressure analysis, relationship dynamics, contradiction, arc evidence, and options for a stuck choice.

Worldkeeper

Track rules and consequences

Candidate entities and facts, world-rule implications, timeline effects, lore gaps, and conflicting evidence.

Continuity reader

Find possible conflicts

Naming, state, knowledge, chronology, travel, setup/payoff, and rule findings with resolvable passage evidence and uncertainty.

Scene partner

Help with the next move

Questions, beat options, scene briefs, continuation drafts, and critiques bounded to the current scene and creative intent.

Revision editor

Run one deliberate pass

Developmental, character, pacing, voice, clarity, or line suggestions. The pass objective constrains what may change.

Research partner

Separate evidence from invention

Organizes writer-provided research at launch. Open-web research is later and requires separate egress, citation, and SSRF controls.

Roles are product language over versioned workflows and capability policy—not distinct autonomous personalities with unrelated memory.

Capability model

Each built-in workflow, local connection, or remote MCP client receives a server-created, project-scoped grant. Client names and self-reported model details are display metadata, not trusted identity. Human author/editor roles and agent grants are related but distinct: an agent never inherits all authority merely because an editor invoked it.

TierCapabilityLaunch posture
0Discover server version, paired project label, and granted capabilitiesNo global project listing
1Read metadata: structure, titles, entity names, counts, revision IDsGrant by project and resource type
2Read selected manuscript, story, notes, and research contentPlain-language disclosure that a client can copy what it reads
3Create noncanonical findings, drafts, candidate facts, and proposalsDefault maximum for external MCP clients
4Apply an already writer-approved, hash-bound proposalFirst-party core path; not advertised to ordinary external clients
5Delete, purge history, manage grants, security policy, or external exportFirst-party UI only, with separate confirmation; later scope

Effective permission is the intersection of server policy, author-issued grant, project role, connection profile, tool contract, and requested resource scope. Egress is orthogonal: content-read permission does not grant permission to send that content to an LLM, website, or export destination.

Proposal lifecycle

Submitted→ Validated / invalid→ Ready→ Accepted / rejected / stale→ Applied / conflicted
  1. Create an immutable context bundle of authorized resource revisions.
  2. Accept model or external-client output as untrusted typed data.
  3. Validate schema, capability, resource scope, limits, evidence, and base revisions.
  4. Stamp server-observed provenance and compute a proposal content hash.
  5. Show authors and editors the outcome, impact, one or more labeled variants, before/after diff, evidence, source scope, provenance, and stale state.
  6. Bind acceptance to the exact proposal hash; accepted subsets become a new child proposal.
  7. Revalidate policy and every expected revision in one transaction.
  8. Apply atomically, create immutable before/after checkpoints, emit audit events, and expose undo.

Drafts have their own noncanonical lifecycle. Moving a draft into a scene creates a proposal; “promote draft” is not a direct write shortcut.

Multiple prose variants

A variant set has one assignment, one immutable base, and two or more independently reviewable child proposals. Each variant states its intended tradeoff—for example “quieter,” “more urgent,” or “preserve structure, sharpen voice.” Combining selections creates a new human-composed child proposal with all source variant IDs; it never splices model output directly into the live draft.

Human suggestions and model proposals may share diff infrastructure, but their origin is never visually collapsed. A human editor’s words are attributed to that editor; an agent result records the initiating human, workflow, provider/model, and final human decision.

Review anatomy

  1. Outcome: what is proposed and why.
  2. Impact: affected scenes, facts, structure, and estimated size.
  3. Diff: additions, removals, moves, metadata, and non-color indicators.
  4. Evidence: anchored, resolvable source passages and story records.
  5. Provenance: writer or client initiator, role, provider/model, workflow version, time, and grant.
  6. Context receipt: exactly what the workflow read and what left the device.
  7. Decision: compare variants, accept selected, combine, edit then apply, keep original, send back, or dismiss.
  8. Safety: base revision, stale/conflict status, and the persistent statement “Your draft has not changed.”

“Accept all” may exist but is visually secondary. Variant comparison preserves the unchanged base beside each option. Applied work remains in history, attributable, and immediately undoable.

UI and MCP parity contract

Maintain a central capability registry used by the responsive web app, collaboration service, platform shells, and MCP:

capability ID
  → core command or query
  → UI action
  → MCP resource or tool
  → required scopes
  → platform exceptions
  → audit events

CI should fail when a shipped writer-facing domain capability has no MCP binding or documented platform exception. Adapter contract fixtures invoke the same use case through UI-facing and MCP-facing adapters with the same project snapshot and grant, then compare normalized result, authorization decision, domain events, audit events, stable error codes, conflict behavior, and idempotency.

Parity is capability equivalence—not permission equivalence. An author may approve, an editor may suggest or compare within scope, and an external client may only submit unless a narrower explicit grant says otherwise.

Prompt injection and exfiltration defenses

Project and imported content

  • Every content block has instructionAuthority: none, even if it is canonical prose.
  • Parse imported content into inert text or validated editor JSON; strip scripts, active HTML, event handlers, remote images, and unsafe links.
  • Never automatically fetch a URL mentioned in manuscript, notes, model output, or MCP content.
  • Tag origin and sensitivity; derived summaries inherit all source restrictions.
  • Render imported and agent output with escaping and safe-link handling.

Runtime enforcement

  • Models receive narrow domain tools only—no environment, credentials, filesystem, shell, or unrestricted network.
  • Call, token, time, artifact-size, and egress budgets are enforced outside the model.
  • Every output is schema-validated; unknown operations and out-of-bundle references fail closed.
  • Permission revocation takes effect on the next request and stops an active run at the next enforced boundary.
  • Provider credentials never enter prompts, resources, errors, audit payloads, diagnostics, or exports.

These controls constrain what a client can do through Ghostwriter. They do not compensate for a compromised account, browser session, collaborator device, or backend operator boundary; identity, session security, encryption, service access, backup, and incident response must be addressed by the shared-backend plan.

Conflict, replay, and destructive safeguards

  • Every operation carries exact base revisions and target hashes. Any mismatch conflicts the proposal and applies nothing.
  • Overlapping pending proposals are identified before review. Launch performs no automatic prose merge.
  • Rebase creates a new proposal with new provenance; it never rewrites the original history.
  • Idempotency keys make repeated submissions and apply requests produce no duplicate effect.
  • Apply creates an automatic pre-apply checkpoint and one-click undo as a new revision.
  • No wildcard targets, hidden cascades, or external MCP deletion tools at launch.
  • Intentional ambiguity, unreliable narration, belief, rumor, dream, and disputed facts must be modeled so continuity checks do not “fix” them into objective canon.

Project audit model

Use an append-only audit stream in the server-authoritative project service without making the entire product event-sourced. Record grant changes, reads and denied reads, context assembly, agent runs, egress, proposal transitions, apply/undo, schema failures, and security violations.

Events include project, sequence, actor kind, server-observed profile, action, capability, allow/deny, resource revision references, grant and policy version, correlation/idempotency IDs, proposal/run IDs, before/after revisions, provider/model metadata where relevant, egress destination/classification/byte count, and outcome.

Do not record raw manuscript text, credentials, approval secrets, full prompts, or provider responses by default.

Verification and long-form evaluation

Deterministic safety suite

  • Authorization matrix for every actor, capability tier, resource class, and operation.
  • Property: no agent command can emit a canonical revision event.
  • Property: apply requires matching writer approval, proposal hash, capability, and every base revision.
  • State-machine, stale revision, overlap, replay, idempotency, atomicity, crash recovery, and cross-project leakage tests.
  • Schema fuzzing, oversized payloads, malformed URIs, Unicode edge cases, archive traversal, unsafe HTML, and malicious links.
  • Audit completeness plus assertions that content and credentials never enter logs.
  • Prompt-injection fixtures must cause zero unauthorized tool calls, writes, reads, or egress.

Original synthetic story benchmark

Build a copyright-safe 80,000–120,000-word test novel with a gold fact ledger and timeline. Seed contradictions at short and 60,000+ word distances; aliases, object state, ages, locations, travel time, world rules, knowledge leaks, and relationship changes; intentional lies, dreams, flashbacks, rumors, unreliable narrators, and nonlinear structure; multiple POV voices; structural setups/payoffs; and embedded prompt injections, encoded instructions, tool-like JSON, hidden markup, and malicious URLs.

PR CI uses deterministic fake model outputs. Provider-dependent evaluations run three times against pinned model and workflow versions before release and whenever either changes; quality uses the median, while safety must pass every run.

Launch and later gates

Capability registry, truth classifications, revisions and hashes, proposal state machine, evidence validation, deterministic policy, transactional apply/undo, audit model, security ADR, and threat model.

Web agentic launch

Bounded built-in workflows, variant review, and paired MCP

Tiers 0–3, visible context receipts, one provider decision, author/editor review roles, multi-variant comparison, revocation, provenance, parity contracts, adversarial suite, and long-form quality gates. Remote MCP is included only if the shared-project authorization ADR covers it; arbitrary web research remains out.

Remote access only when a real workflow needs it

OAuth resource-bound tokens, PKCE, exact redirects, TLS, origin validation, DNS-rebinding protection, SSRF-safe research gateway, stronger client identity, encrypted traces/storage, signed updates, and continuous red-team evaluation.
