Intent

Define how Ghostwriter supports the complete life of a novel—from spark or imported draft through worldbuilding, full-book drafting, editorial review, revision, and named editions—and scales its connected story knowledge across multi-book arcs. Experienced and aspiring novelists share one artifact model with different levels of optional guidance.

The primary product is a real-time web workspace that combines a rich manuscript editor with a spatial Story Canvas, notes, Git-inspired content history, alternate versions, tracked editorial work, and project-aware assistants. Agents may draft ideas, edit passages, and propose multiple prose variants, but always work in tandem with authors and authorized editors.

The output of this effort is a decision-ready blueprint and phased plan portfolio—not production implementation. The near-term product is the trusted creation environment. Publisher, studio, rights, discovery, commerce, and marketplace systems remain a long-horizon option that must not distort the authoring core.

Acceptance criteria

  • An aspiring novelist can follow one documented path from raw idea to an editable first scene without needing to understand writing jargon or prompt engineering.
  • An experienced novelist can import or begin blank, navigate a full book and shared series knowledge directly, use keyboard-first actions, and disable guidance.
  • The domain and information architecture support a project containing multiple ordered books with project-wide characters, world rules, timeline, notes, and cross-book arcs.
  • The Story Canvas and manuscript use the same scenes and story objects, supporting write-first, storyboard-first, split-view, and derivation workflows without duplicate truth.
  • Git-inspired workflows cover autosave, checkpoints, alternate versions, comparison, restoration, attribution, review requests, and named editions without exposing Git mechanics.
  • Authors and editors can collaborate through explicit roles, comments, tracked suggestions, version review, and accountable attribution in the primary web experience.
  • The blueprint names the canonical project artifacts and visibly distinguishes authored text, planned intent, confirmed canon, inferred findings, and pending proposals.
  • Every AI or external-agent mutation, including multiple prose variants, follows an inspectable propose → compare → review → apply → undo path; no model receives silent canonical-write authority.
  • Core UI capabilities have corresponding core use cases and a documented MCP binding or an explicit, justified platform exception.
  • The technical blueprint preserves Tiptap, the platform-neutral core, and MCP parity while making responsive real-time web the primary product and treating wrappers as conveniences.
  • Storage authority, offline behavior, real-time merge, provider, credential, security, and marketplace choices are separated into accepted requirements, recommendations, spikes, and founder decision gates.
  • The roadmap decomposes the vision into small candidate plans with dependencies, scope boundaries, tests, measurable exit gates, and intentionally deferred work.
  • Visual mockups cover the primary wide-web workflow, Story Canvas 2.0 direction, focused narrow-web workflow, and supplied Ghostwriter brand lockup.
  • The plan package links cleanly, uses semantic HTML, remains readable without JavaScript, and does not break pnpm verify.

Tests and verification

  • Journey walkthroughs: trace “spark to first scene,” “import and organize a draft,” “review an agent edit,” “resolve a continuity issue,” and “capture on mobile, review on desktop.”
  • Trust audit: every generated artifact has a visible authority state, source scope, provenance, review action, stale-state behavior, and undo path where it can affect canonical work.
  • Parity audit: roadmap slices identify core, UI, MCP, and platform-adapter work together; exceptions are explicit.
  • Decision audit: each durable technical recommendation maps to an ADR candidate and identifies when the decision must be made.
  • Responsive review: inspect mockups at approximately 1440 × 960 and 390 × 844, including connected, reconnecting/recovery, empty, running, ready, stale, conflicted, and failed states.
  • Repository check: run pnpm verify before handoff and inspect all changed HTML links and document structure.
  • Future validation: test the proposed journeys with 12–15 target writers before treating product assumptions as validated requirements.

Documentation and ADR impact

This plan produces recommendations only. After founder review, update docs/PRODUCT.md with the approved positioning, users, principles, and v1 boundary, and update docs/ARCHITECTURE.md with accepted package and data-flow decisions.

Likely ADRs, each created in the implementation plan that first needs it:

  1. Project truth model and scene-granular document boundaries.
  2. Canonical editor format, stable block IDs, revisions, and provenance.
  3. Concrete shared backend, minimal browser recovery, backup, account exit, and portable project archive under ADR 0002.
  4. ADR 0002 accepts server-authoritative online-only projects and editorial collaboration without same-scene co-editing; agent mutation policy still needs its own ADR.
  5. AI provider, credentials, context egress, and web runtime boundary.
  6. Backup, identity, sync, and collaboration architecture—only when demanded.

Risks and decisions

Accepted

Serve novelists and aspiring novelists together

Lead with the complete novel-writing experience for full books and multi-book arcs. Guidance is optional inside one project model rather than a separate “generate my book” product.

Accepted

Make the web workspace live and collaborative

The rich editor, Story Canvas, notes, history, comments, tracked suggestions, editors, and assistants operate on one shared project available wherever a browser is convenient.

Accepted

Use Git ideas without making writers use Git

Expose checkpoints, variants, compare, restore, attribution, review, and named editions through writer language and a rich visual interface.

Accepted

Keep agents in tandem with humans

Assistants can brainstorm, draft, edit, and prepare multiple prose variants. Authors and authorized editors compare, combine, revise, accept, or reject them.

Accepted · ADR 0002

Use server-authoritative online-only v1 projects

The browser keeps minimal unacknowledged-work recovery, not a complete offline replica. Project updates, presence, comments, tracked suggestions, and version review are live; only one collaborator directly edits a scene body at a time.

Deferred

Marketplace and industry architecture

Do not build discovery, licensing, rights, payments, social, or studio tenancy until creation retention, finished exports, and real sharing demand prove the entry conditions.
