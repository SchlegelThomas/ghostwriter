Ghostwriter delivery plan · Phase 1.2 (backend)

Lakebase backend, migrations, and branch-based CI/CD

Status: active
Branch: feat/multi-book-domain-kernel (shared with the kernel slice per founder direction)
Owner: founder + agent

Intent

Stand up the durable, server-authoritative persistence and delivery pipeline required by ADR 0002: a Databricks Lakebase Postgres database behind the core ProjectRepository port, version-controlled migrations, a thin Node backend service, and CI/CD that gives every pull request an isolated Lakebase database branch. This turns the in-memory domain kernel into a real backend without changing domain ownership rules.

Google OAuth accounts and profile creation, subscription/billing, real-time transport, and browser recovery are explicitly out of scope and follow in their own plans once this backend exists. This plan does not change packages/core ownership rules, add offline/CRDT behavior, or introduce a second canonical store.

Acceptance criteria

  • ADR 0004 records the Lakebase database, Drizzle migrations, Node/Hono runtime, and branch-per-PR CI/CD decisions.
  • packages/storage implements the core ProjectRepository port against Postgres with Drizzle, enforcing the ADR 0003 normalized schema and whole-project validation inside write transactions; no provider or SQL types enter packages/core.
  • Version-controlled Drizzle migrations exist under packages/storage/drizzle/, are applied by pnpm db:migrate, and reproduce the schema from empty.
  • Repository and backend tests run against in-process Postgres (PGlite) so pnpm verify passes with no live database or cloud credentials.
  • apps/backend serves health and the project navigator read model from Postgres using the same core services as the UI and MCP, and is container-deployable.
  • CI runs migrations and the storage/backend suites on every PR; a separate workflow provisions a Lakebase branch per PR, migrates it, and deletes it on close.
  • A deploy workflow migrates production and deploys the backend on merge to main.
  • Operations and architecture docs describe the database, migration, branch lifecycle, secrets, and deploy flows.

Tasks

  1. Record the backend/database/CI decisions in ADR 0004 and this plan.
  2. Create packages/storage: Drizzle schema for the multi-book model, client factories (node-postgres + PGlite), migrator, seed, and a Postgres ProjectRepository adapter with parity validation.
  3. Generate and commit the initial migration; add repository contract tests against PGlite.
  4. Create apps/backend: Hono app and server wiring core services to the Postgres adapter, config from environment, health and navigator routes, and hermetic tests.
  5. Add Lakebase branch/credential automation (CLI-driven scripts) and root db:* scripts.
  6. Add CI database checks, the PR branch lifecycle workflow, and the backend deploy workflow.
  7. Update operations, architecture, README, and the handoff; run full verification.

Tests and verification

  • Storage unit/contract: create project + book, read navigator projection, ordering, cross-record rejection, and transaction rollback against PGlite with real migrations applied.
  • Migration integrity: migrations applied from empty produce a schema the adapter round-trips; drift is caught in CI.
  • Backend: in-process requests to /health and the navigator route return the same projection the core produces; unknown project yields 404.
  • Pipeline (manual/CI): a PR provisions and migrates a Lakebase branch and tears it down; merge migrates production and deploys the backend.
  • Repository: pnpm verify plus targeted storage/backend suites before handoff.

Living design source: not applicable — this slice is backend/infrastructure with no product-facing UI change. The living design continues to govern client work.

Risks and decisions

  • Founder-accepted: Databricks Lakebase (managed portal), Node/Hono runtime, Drizzle migrations, branch-per-PR CI/CD, shared branch with the kernel slice, and available Databricks credentials for CI.
  • Lakebase branch/endpoint APIs are Beta; command shapes may change. They are isolated to scripts and workflows so updates stay contained.
  • Live provisioning requires Databricks and deploy-host secrets that only the founder can set. Code and pipelines are built to run hermetically (PGlite) until those secrets are added, then activate.
  • OAuth token TTL (~1h) requires per-connection refresh for long-lived services; handled in the connection layer, not core.
  • Identity, profiles, subscriptions, real-time transport, and browser recovery remain open and are not decided here.
