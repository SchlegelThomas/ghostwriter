Where we left off

Last updated: 2026-07-11 · Current branch: main

Current state

PR #4 is merged to main. Ghostwriter now has a real, deployed backend: a normalized multi-book domain kernel; a responsive brand-aligned fixture navigator and MCP tool over the same core query; a Drizzle Postgres ProjectRepository adapter; and a Node/Hono service running on Fly.io backed by Databricks Lakebase. CI runs hermetic tests (PGlite), gives every PR its own Lakebase database branch, and deploys the backend on merge. Editing, collaboration, Story Canvas, auth, and AI are still not implemented.

Live: backend at https://ghostwriter-backend.fly.dev (/health ok; DB-backed route authenticates to Lakebase via the CI/app service principal). Lakebase project projects/ghostwriter (branch production, endpoint primary, AWS us-east-2, PG 17) is provisioned and migrated.

Committed foundations

  • Responsive real-time web is the primary product; Expo remains the universal client and wrappers add conveniences.
  • Tiptap (ProseMirror) editor, hosted by Expo DOM components on native.
  • GitHub Actions CI/CD; Cloudflare Pages direct upload; protected main.
  • Full novels and multi-book projects share story knowledge and cross-book arcs.
  • Story Canvas and Draft are linked views of the same scenes and story objects.
  • Authors and editors use attributable comments, suggestions, variants, compare/combine, restore, and named editions.
  • Agents propose ideas, edits, and multiple prose versions; humans decide what becomes canonical.
  • ADR 0002 sets online-only, server-authoritative v1 projects with minimal unacknowledged-work recovery and one direct scene editor at a time; full offline and same-scene multi-cursor editing are deferred.
  • ADR 0003 sets normalized project, book, scene, story-knowledge, and edition boundaries; manuscript and Canvas views reference the same stable scene records.
  • ADR 0004 selects Databricks Lakebase Postgres, Drizzle migrations, a Node/Hono backend, and a database branch per pull request as the server-authoritative store.
  • Writer ownership through complete history, export, account exit, platform-neutral core, and MCP parity remains non-negotiable.

Active plan

The only remaining active plan is the product/architecture blueprint: agentic writing experience, which retains the experience, technical, agent, roadmap, and visual artifacts plus the still-open platform decisions below. Living design source: Ghostwriter Mockups 2.0 (never archived).

Incomplete and awaiting founder review

The product direction is substantially accepted and the backend topology is now decided and deployed. These material decisions remain open:

  1. Identity/authentication (Google OAuth accounts + profile creation) on top of the Lakebase backend.
  2. Subscription/billing management.
  3. Real-time transport, presence, scene leases, backup/retention/deletion, and remote MCP authorization (backend/database selected in ADR 0004; live-update layer still open).
  4. Initial import/export formats; browser access is the accepted primary portability experience.
  5. AI provider, credentials, web egress, and local-model posture.
  6. Confirmation that marketplace, rights, payments, and studio systems remain outside creation v1.

Recommended next step

Start the next plan for Google OAuth accounts and profile creation (branch feat/<slug>, new ADR for identity), then subscription management. Both build on the deployed Lakebase backend. Keep the real-time transport, formats, AI-provider, and marketplace items open until their plans need them.
