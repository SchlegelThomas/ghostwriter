Ghostwriter delivery plan · Phase 1.1

Build the multi-book domain kernel

Status: active
Branch: feat/multi-book-domain-kernel
Owner: founder + agent

Intent

Replace the scaffold manuscript example with the first trustworthy Ghostwriter product slice: a platform-neutral model for projects containing one or more ordered books, manuscript structure, scenes, and project-wide story knowledge. Expose the same read model through a responsive fixture navigator and MCP so later persistence, editing, collaboration, and Story Canvas plans build on one domain boundary.

The fixture navigator must use Ghostwriter Mockups 2.0 as its visual and interaction source, including the supplied packages/ui/src/Ghostwriter.png lockup and the living design's typography, color, rail, topbar, and inspector language. This slice uses deterministic in-memory fixtures. Shared persistence, identity, networking, browser recovery, Tiptap scene bodies, revisions, collaboration, Story Canvas editing, import/export, and AI are explicitly out of scope and remain governed by the accepted roadmap and ADR 0002.

Acceptance criteria

  • The core represents a story project with one or more ordered books, book-owned manuscript structures, ordered chapters, scene metadata, editions, and project-wide story knowledge references without duplicating scene truth.
  • Creation and update functions reject empty identity or title fields, duplicate IDs, cross-project or cross-book references, and manuscript structures that reference unknown scenes.
  • Core transitions are immutable and platform-neutral; no React, DOM, Expo, Node, MCP, persistence-provider, or transport types enter packages/core.
  • Plain command/query services depend on explicit repository, ID, and clock ports, and a memory repository proves the contracts without choosing the shared backend.
  • A capability registry identifies the shipped project-read capability and its core, UI, and MCP bindings.
  • The responsive client shows a clearly labeled fixture project with multiple books and a navigable manuscript/scene hierarchy derived from core queries.
  • The wide-web client chrome follows the living 2.0 design source: supplied logo lockup, Parisienne/Jost/Cormorant brand typography, 52px topbar, 58px project rail including Inbox, contextual navigator, warm writing surface, and 276px scene-context inspector.
  • Fixture-only content remains honest and visually secondary: no metric dashboard, save-state, collaboration, Canvas, editor, or MCP implementation claim appears as shipped product behavior.
  • The MCP server exposes the same fixture project summary and hierarchy through a read-only tool or resource backed by the same core query.
  • Core tests cover invariants, immutability, ordering, memory repository behavior, and UI/MCP parity metadata; pnpm verify passes.

Tasks

  1. Record the project, book, manuscript, scene, edition, story-knowledge, identity, and aggregate boundaries in an ADR.
  2. Split the scaffold core into focused domain, port, use-case, capability, fixture, and public-export modules while preserving strict ESM and platform neutrality.
  3. Implement immutable constructors/transitions, validation errors, ordered references, repository contracts, query services, and an in-memory adapter.
  4. Add representative original-fiction fixtures for a multi-book project and test all domain and repository invariants.
  5. Replace the Expo welcome placeholder with an accessible responsive fixture navigator that consumes only the public core query/read model.
  6. Review the living 2.0 design directly, add shared brand/theme tokens and fonts, use the supplied logo, and align the fixture shell without implementing deferred Story Canvas behavior.
  7. Replace the MCP scaffold status payload with a read-only project hierarchy capability backed by that same query/read model, and add a process-level smoke test if practical in this slice.
  8. Update architecture/API documentation, the active record log, and the handoff; run targeted checks and full repository verification.

Tests and verification

  • Core unit: valid multi-book creation, stable order, immutable updates, duplicate/unknown/cross-book reference rejection, and query projection.
  • Repository contract: save/get/list behavior and defensive isolation against caller mutation for the memory adapter.
  • Parity: capability metadata names the core query and both shipped bindings; UI and MCP fixture projections equal the core result.
  • MCP integration: launch the stdio server, list or invoke the project-read capability, validate structured output, and shut down cleanly.
  • Manual web: inspect the fixture navigator at approximately 1440 × 960 and 390 × 844 for readable hierarchy, supplied logo visibility, living-design typography/colors, complete rail labels, honest fixture state, and graceful responsive collapse.
  • Repository: run targeted Vitest suites during implementation and pnpm verify before handoff.

Documentation and ADR impact

Create ADR 0003 for the multi-book aggregate and reference boundaries. Update docs/ARCHITECTURE.md only where implementation makes the accepted blueprint concrete. Keep docs/PRODUCT.md unchanged unless implementation reveals a product conflict. Update this plan, its record log, and plans/WHERE-I-LEFT-OFF.html with delivery evidence.

Source direction: delivery roadmap, technical blueprint, and living design source. The living design governs current product chrome and interaction direction; the plan-scoped mockups are historical context where they conflict.

Risks and decisions

  • The domain must establish stable ownership and reference boundaries without pre-deciding a database schema, real-time protocol, editor document schema, or revision graph.
  • Project-wide story knowledge in this slice is a typed reference envelope and authority state, not the full story-bible model scheduled for Phase 2.1.
  • Edition records establish identity and naming only; immutable revision heads and compare/restore behavior remain in the revisions plan.
  • The UI is an honest fixture navigator, not a persistence-backed product claim. It must label sample data and avoid implying save, collaboration, or offline behavior.
  • The supplied PNG is a large-format lockup with a light background. Use it intact in a suitable branded surface and use the living design's text wordmark in compact chrome; do not invent an unofficial transparent icon crop in this slice.
  • Concrete backend, identity, retention, recovery, AI, and import/export choices remain open and are not silently selected here.
