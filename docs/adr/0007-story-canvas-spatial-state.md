# 0007: Story Canvas spatial state and manuscript authority

- Status: accepted
- Date: 2026-07-12
- Plan: `plans/active/2026-07-11-authenticated-project-crud/plan.html`
- Related: ADR 0002 (server authority), ADR 0003 (project and scene boundaries), ADR 0006 (proposed scene history)

## Context

Ghostwriter's living design makes Story Canvas and Draft two views of one project. Writers may begin
spatially or in prose, but both journeys must converge on the same scene and story records. Canvas
placement must not copy scene bodies or silently become another manuscript order. The design also
requires regions, notes, typed links, provisional ghost objects, imagery metadata, a manuscript-
derived reading-order spine, wide split view, and a non-spatial accessible path.

The current kernel has canonical scene IDs, manuscript order, scene metadata, and story-knowledge
links but no Canvas persistence. Using the coarse project version for every drag would conflict with
unrelated metadata commands. Persisting zoom, selection, and canonical object placement together
would also confuse shared story state with personal view preferences.

## Decision

### Ownership and records

- A project owns one canonical Canvas board. The board has its own monotonically increasing version
  independent from project metadata and scene document versions.
- Canonical `canvas_objects` support scene cards, story-knowledge cards, notes, regions, and image
  references. Scene/story cards reference stable canonical IDs and never copy prose or story-
  knowledge bodies.
- Canonical `canvas_links` connect Canvas object IDs with a typed relationship such as pin, thread,
  beat, dependency, or reference. Links and objects carry `confirmed` or `provisional` authority;
  dismissal records enough provenance to avoid immediately resurfacing the same deterministic
  suggestion.
- Spatial state includes x/y position, size, z-order, optional region membership, label/style
  metadata, and archive state. Image references contain metadata and an optional future asset ID,
  not arbitrary fetched URLs or generated binaries in this epic.
- Pan/zoom viewport and last-open selection are per-account view preferences, stored separately
  from canonical board state. Current selection remains transient client state shared by Draft,
  Canvas, and split view.

### Commands, concurrency, and history

- Canvas create/place/move/resize/link/unlink/confirm/dismiss/archive/restore commands require owner
  authorization and `expectedCanvasVersion`; each accepted gesture commits once on pointer/key
  completion rather than writing every pointer movement.
- The board uses relational current-state tables for efficient projection. Meaningful manual,
  sustained-idle, restore, and project-checkpoint boundaries create content-addressed immutable
  Canvas snapshots; immediate Undo issues a guarded inverse command. Canvas is not event-sourced.
- A stale board command applies nothing and reloads the latest board. Canvas conflicts never mutate
  scene prose or manuscript structure.
- Canvas-created scenes use one application transaction that requires explicit book/chapter/
  unassigned manuscript placement, invokes the canonical scene-creation path, initializes its scene
  document, and places a Canvas card referencing the resulting scene ID.

### Manuscript authority and reading-order spine

- Canonical reading order is derived only from the ordered book/part/chapter scene references in the
  manuscript kernel. Canvas coordinates, regions, and links do not reorder Draft.
- Scene cards may carry an explicit Canvas story-order hint. The reading-order spine compares that
  hint with canonical manuscript order and exposes drift; applying a reorder requires a separate
  explicit manuscript command and confirmation.
- Rename, archive, restore, or move of a canonical scene is reflected by ID in Canvas projections.
  Archived or missing references become clear stale-link states and are never silently dropped.

### UI and accessibility

- Wide web supports spatial Canvas, Draft, and split view with one selected scene ID and preserved
  inspector/link context. The founder-authored living mockup governs visual language and is not
  regenerated by this ADR.
- Narrow web supports reading, review, capture, spine navigation, and canonical actions through an
  ordered representation; it does not compress freeform pan/zoom/drag into a phone layout.
- Every drag action has a keyboard and screen-reader equivalent through the ordered Outline/spine
  view. Focus order, labels, coordinates, regions, links, authority, and stale state are available
  without relying on visual position or dashed styling alone.
- Writer-created objects and links are confirmed/solid. Deterministic fixture ghosts are
  provisional/dashed and labeled until confirmed or dismissed. This epic makes no model calls.

### Scope boundary

- The Canvas foundation includes state management, spatial/ordered projections, links, regions,
  notes, image metadata, provisional fixtures, spine drift, split-view selection, and persistence.
- It excludes image generation/hosting, automated manuscript derivation, real AI inference,
  collaboration presence, simultaneous object editing, full mobile freeform Canvas, marketplace
  assets, and canonical MCP writes.

## Options considered

- **Store Canvas placement on scene metadata only** — rejected because notes, regions, imagery, and
  links are project-scoped objects and would overload scene records.
- **Make Canvas order canonical manuscript order** — rejected by ADR 0003 and the living design;
  spatial meaning must not silently reorder prose.
- **Use one project version for all Canvas gestures** — rejected because spatial interaction would
  create false conflicts with book, metadata, and story-knowledge commands.
- **Persist a single Canvas JSON blob** — simple initially, but weak for reference integrity,
  partial updates, indexing, 500+ objects, stale links, and accessible projections.
- **Event-source every pointer movement** — rejected as noisy and expensive. Commit completed
  gestures, use guarded inverse commands for immediate Undo, and checkpoint meaningful snapshots.
- **Ship only a visual drag surface** — rejected because every canonical action needs an equivalent
  keyboard and screen-reader path.

## Consequences

- Canvas and Draft share canonical scene/story IDs while preserving separate, appropriately scoped
  concurrency.
- The database gains board, object, link, preference, dismissal, and immutable snapshot records plus
  indexes for project, reference, region, authority, and archive state.
- Scene creation from Canvas becomes a coordinated application transaction across metadata,
  document initialization, and board placement.
- The client needs shared selection state, wide spatial rendering, viewport-aware culling, an
  accessible ordered view, and narrow read/review behavior.
- Manuscript order remains unambiguous; drift is visible and applying a reorder remains explicit.
- Real-time collaboration, AI derivation, binary assets, and marketplace concerns remain separate
  future architecture decisions.
